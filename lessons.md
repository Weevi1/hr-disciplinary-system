# Lessons — File by FIFO

> Patterns learned from corrections. Reviewed at session start.
> Format: `[date] What went wrong → What to do instead`

## Security
- [2025-10-02] `isAuthenticated()` used 30+ times as security bypass in Firestore rules → Always validate role/organization membership explicitly. Never use `allow read: if isAuthenticated()` as a catch-all
- [2025-10-02] `allow read, list, write: if true` found in rules → Never use `true` in security rules. Always scope to org membership + role
- [Session 64] `file.makePublic()` used for evidence files → Use signed URLs with expiry instead: `getSignedUrl({ expires: Date.now() + 15*60*1000 })`
- [2026-05-08] `auth-export.json` + `users.json` (real Firebase Auth exports with emails + bcrypt hashes) committed to git in 2 historical commits → Scrubbed via `git filter-repo --invert-paths --path auth-export.json --path users.json --force` + force-push to GitHub + `git fetch && reset --hard + git gc --aggressive --prune=now` on cats. Pre-scrub mirror saved at `cats:/storage/backups/git-history-scrub/hr-mirror-pre-scrub-2026-05-08_1014`. Lesson: NEVER commit Firebase Auth exports, Firestore data exports, or any file with real emails/hashes/IDs. Always `gitignore *.json exports + test data dumps from the start` (now in `.gitignore`: `auth-export.json`, `users.json`, `*-export.json`). Verify `.gitignore` before the first commit on any new file class.
- [2026-05-08] Storage rules had wide-open catch-all `match /{allPaths=**} { allow read, write: if request.auth != null; }` AND `userOrgIndex/{userId}` allowed any authenticated user to write any other user's index entry → Replaced catch-all with explicit `organizations/{orgId}/{allPaths=**}` rule (org-membership / super-user / reseller checks) + deny-by-default catch-all + new `warnings/{orgId}/{warningId}/pdfs/{file}` rule (was relying on the old catch-all). Tightened `userOrgIndex` write to `request.auth.uid == userId`. Lesson: when a "fix" tightens a load-bearing catch-all, scan the storage bucket for ALL paths in actual use first (`bucket.getFiles()` admin SDK script) — bucket reality beats grep, several paths only relied on the catch-all and would have silently broken otherwise.

## Auth & Users
- [2025-10-23] New users had to sign out/back in to get their role → Always call `auth.setCustomUserClaims()` immediately after creating user doc. Applied to all 5 creation functions
- [Session 63] Reseller user creation missing `uid` field, role saved as raw string → Always set `uid: userRecord.uid` and normalize role to object format `{ id, name, level }`
- [Session 63] `issuedBy` empty on warnings when user has no `uid` → Provide fallback: `user?.id || 'unknown'`
- [2026-05-10] Phase 1.5 dual custom-claim format cleanup: codebase historically supported both NEW abbreviated (`r`/`org`/`res`) and LEGACY verbose (`role`/`organizationId`/`resellerId`) formats with `||` fallbacks scattered across rules + middleware + frontend. Migration order: (1) update remaining LEGACY setters to NEW format, (2) bulk-refresh production users via `customClaims.refreshOrganizationUserClaims()` + direct Admin SDK script (9 users, 1 migrated, 8 already clean), (3) remove fallbacks. Single canonical format now enforced. Lesson: when introducing an abbreviated/optimised claim shape, plan the cutover from day one — track all setters and readers, write both formats only during transition, then commit to a single format. Dual-format support doubles every permission check and rots over time.
- [2026-05-10] Pre-existing bug discovered during Phase 1.5: `frontend/src/auth/AuthContext.tsx:221-222` was reading `claims.orgId` (a field that exists in NEITHER format — neither NEW `org` nor LEGACY `organizationId`) before falling back to `organizationId`. Effectively the optimistic org-fetch always missed the NEW format and only worked via legacy fallback. Lesson: when implementing dual-format readers, write a regression test that asserts the NEW path is actually exercised. Otherwise typos hide for years.
- [2026-05-11] Phase 2 Tier 2C smoke test surfaced a latent role-format inconsistency on the super-user account: custom claim was `r: "superuser"` (no hyphen) and `users/{uid}` doc had `role: "superuser"`, but the canonical `UserRoleId` type is `"super-user"` (with hyphen) and Firestore rules enforce `request.auth.token.r == "super-user"`. Phase 1.5's legacy `|| role` fallback removal surfaced this — the user could no longer load orgs or resellers. Phase 1.5's bulk-refresh function (`customClaims.refreshOrganizationUserClaims()`) iterates *org users* and never touched the super-user. Cloud Functions also read role from the user *document* (`userData?.role?.id || userData?.role`) not just the JWT, so both stores needed patching. Lesson: any role-format migration must (a) iterate ALL users including super-users and any non-org-scoped accounts, AND (b) patch both the Auth custom claim AND the Firestore user doc — Cloud Functions and Firestore rules read from different stores. Add a CI sanity check that asserts every active user's claim role and doc role are both members of `UserRoleId`.

## Modals & UI
- [Session 21] Body scrolling behind modals → Use `usePreventBodyScroll` hook. Never `position: absolute` — use `fixed inset-0`
- [Session 59] SignaturePad canvas sizing broken inside modals → Use React Portal to escape parent CSS, inline styles only, `useLayoutEffect` for viewport-based sizing
- [2026-06-03] Declared "scroll fix shipped" 3× while user kept showing screenshots of broken scroll. Type-check passed, dev server hot-reloaded, but the actual feature was still broken → "Build clean" ≠ "feature works." For any UX/CSS/layout change, do NOT claim it works after Vite build alone — ask the user to verify in the browser before declaring victory, OR use a screenshot/browser-driven verification tool. The harness can't tell visual rendering passed; only the user (or playwright) can.
- [2026-06-03] Modal scroll trap: Tailwind `flex-1 min-h-0 overflow-y-auto` inside a `max-h-[90vh] flex flex-col` parent silently fails to engage scroll in Chrome — the flex child can't compute a definite height when the parent only has a *max* constraint. Content overflows invisibly, footer (with `flex-shrink-0`) stays pinned, middle just gets clipped → Use `h-[90vh]` (definite height) on the modal wrapper AND inline `style={{ flex: '1 1 0%', minHeight: 0, overflowY: 'auto' }}` on the scroll container. Add `overscrollBehavior: 'contain'` so wheel events don't escape to the page behind the backdrop.
- [2026-06-03] V2 wizard's "Before you continue" notices were invisible to the user → `style={{ backgroundColor: 'var(--color-alert-info-bg)' }}` resolves to `#eff6ff` (light blue), which is functionally indistinguishable from `#ffffff` (white modal). User reported "items are cut off" but they were actually rendered, just camouflaged. Lesson: for notice/alert cards on a white modal, always pair a light background with an explicit `border` — or use solid contrast colors. Don't trust faint CSS-variable backgrounds to convey presence.
- [2026-06-03] V2 wizard initially wrapped its phase content in `<div className="enhanced-warning-wizard-container ...">` to "match V1 structure" — but that class has dozens of `!important` descendant rules in `modal-system.css` (lines 1645–2360) designed for V1 internals. Parallel V2 components should NOT inherit V1 marker classes; they bring along CSS baggage that's invisible until something breaks unexpectedly. Lesson: when building a parallel V2 component, give it its own self-contained styling and skip V1's selector hooks even when it looks cosmetic.

## Warning Delivery
- [2026-06-03] WhatsApp delivery added to wizard Step 5 → WhatsApp's click-to-chat deep link (`wa.me/<digits>?text=…`) can target a specific number AND pre-fill text, but CANNOT pre-attach a file. Don't promise a PDF attachment. Instead send the durable respond-link (`file.fifo.systems/respond/{token}` from `generateResponseToken`) in the message text — the employee taps it to view/download the PDF and respond/appeal. The temp QR link (`TemporaryLinkService`, 1-hour) is wrong for WhatsApp because the recipient may open it days later; the respond token defaults to 30 days and now takes an optional `expiryDays` (WhatsApp passes 180). Number is normalized via the shared `utils/phone.ts` (`toWhatsAppNumber` strips to digits-only). Delivery is recorded only after the manager attests "I've sent it" (browser can't confirm WhatsApp actually sent) — avoids the perpetual-`pending` bug the old stub had.
- [2026-06-03] Self-finalizing delivery panels (email, WhatsApp) must be excluded from the generic DELIVERY-phase Finalize button → `isPhaseValid` for `PhaseV2.DELIVERY` returns false for `email`/`whatsapp` so the shared Finalize stays disabled; each panel writes its own `deliveryMethod`/`deliveryStatus`/`deliveryHistory` + triggers the celebration. `handleFinalize` early-returns for those methods. When adding a new delivery method that completes inside its own panel, add it to BOTH guards or it'll double-record. (Printed joined this set on 2026-06-04.)
- [2026-06-04] "Printed" delivery means the employee collects a physical copy from HR — NOT instant delivery → wizard records `deliveryStatus: 'awaiting_collection'` + emails HR via `notifyHRPrintedCollection`; HR completes the hand-over later from `ReviewDashboard` ("Deliver" button → `EnhancedDeliveryWorkflow` → `PrintDeliveryGuide`: print → hand over with date/location/witness → file). Most of the HR side already existed — don't rebuild it; the wizard side + the completion write were the gaps. The HR collection workflow regenerates the PDF on demand (`PrintDeliveryGuide.generatePDFForWarning`), so no need to pre-store it for printed.
- [2026-06-04] `ReviewDashboard` HR-side delivery completion wrote `status: 'delivered'` (the warning *lifecycle* field) but never `deliveryStatus` → the undelivered queue + badges read `deliveryStatus`, so HR-delivered warnings stayed stuck "Pending Delivery" forever and the guide's captured proof (location/witness/notes) was silently discarded. Fix: write `deliveryStatus: 'delivered'` + `deliveredBy` + append a `deliveryHistory` entry; stop clobbering `status` (`'delivered'` isn't even a valid lifecycle value). Lesson: the warning has TWO status fields — `status` (lifecycle: issued/acknowledged/appealed/…) and `deliveryStatus` (delivery: pending/awaiting_collection/delivered). Delivery code must write `deliveryStatus`; mixing them up is invisible until you check the queue filter.
- [2026-06-04] Adding a new delivery status value is safe here → the codebase only does `=== 'delivered'` / `!== 'delivered'` equality checks on `deliveryStatus`, no switch statements. Added `'awaiting_collection'` to the `DeliveryStatus` union in `types/core.ts` (both the type and the inline `Warning.deliveryStatus` union) with zero downstream breakage. Final collected state stays `'delivered'` so existing filters keep working.

## Lint & Tooling
- [2026-05-11] `npm run lint` was silently broken — `eslint.config.js` had `no-console: ['error', { allow: [] }]` (invalid in ESLint 9; `allow` must be ≥ 1 item) AND no TypeScript parser configured, so `.ts`/`.tsx` files failed to parse before any rule ran → When inheriting an ESLint config, always run `npm run lint` once before trusting it. A green CI doesn't mean lint works; it might mean lint never ran. Fix: wire `typescript-eslint` parser, drop empty `allow: []` (use `'no-console': 'error'` directly), and exempt `src/utils/logger.ts` + test/script globs at the file-pattern level
- [2026-05-11] `Logger.error('msg', error, contextObj)` silently dropped debug context → Logger's third arg is typed `LogContext` (`{ organizationId?, userId?, component?, action?, sessionId? }`) and `formatContext()` only reads those known keys. Passing `{ warningData, organization }` as `context` is silently discarded. When you have unstructured debug payload alongside an error, bundle it into the `error` arg: `Logger.error('msg', { error, warningData, organization })` — the second arg is sanitised in prod but preserved in dev
- [2026-05-11] React `rules-of-hooks` violations were hidden because lint never parsed TS files → After the Phase 4 config fix, 12 real violations surfaced in 3 components (`DashboardRoleSelector`, `AudioPlaybackWidget`, `MainLayout`). All were hooks called *after* an early `return null` / conditional return. Lesson: hooks MUST run unconditionally on every render. If you need a guard, place the early return AFTER every `useState`/`useEffect`/`useCallback`/`useMemo`/`useRef`/`useContext` call. Also: when adopting a new lint rule, re-baseline once — the rule may have been silently dormant
- [2026-05-11] `tsconfig.app.json` `exclude` pattern is the right tool to keep archived migration scripts out of the project tsc baseline → `frontend/src/scripts/_archive/` is excluded explicitly. Archiving (rather than deleting) preserves the script for a future re-shard/backfill while keeping the baseline clean. Pattern: `git mv`, exclude in tsconfig, append a stanza to `_archive/README.md` explaining what it did + how to re-run

## Cloud Functions Runtime + SDK Upgrade
- [2026-05-19] firebase-functions v1 → v2 migration cannot be done in-place on a deployed function — the deploy fails with "Upgrading from 1st Gen to 2nd Gen is not yet supported." Have to `firebase functions:delete <names…> --force` first, then redeploy as v2. Brief outage (~3 min) for those endpoints during the gap. Document the affected function names + brief outage window before doing this on a live tenant. The deploy log doesn't proactively warn — only fails at the IAM step.
- [2026-05-19] v2 HTTP endpoints (`onRequest`) do NOT get implicit public IAM on create — only on first deploy of a brand-new function. After delete + recreate (1st→2nd gen migration), the IAM policy step in `firebase deploy` may fail silently for several functions (per-region quota / transient Cloud Run issues). Symptoms: deploy reports `Failed to set the IAM Policy on the Service projects/.../services/X`, and unauthenticated callers get 403 even on CORS preflight (`OPTIONS` returns 403 instead of 204). Fix: add `invoker: 'public'` to the function options AND run `gcloud run services add-iam-policy-binding <fnname-lowercase> --region=us-central1 --member=allUsers --role=roles/run.invoker` for each affected endpoint as a belt-and-braces measure. The deploy doesn't re-apply IAM on subsequent runs unless the code hash changes.
- [2026-05-19] **`firebase functions:list` reports CONFIGURED runtime, NOT what's actively serving.** Caught during post-deploy audit: 3 functions (`deliverWarningByEmail`, `uploadResponseEvidence`, `notifyHROnAppeal`) showed as `nodejs22` in the listing, but `gcloud run services describe` revealed the active serving revisions were 9 days old on Node 20. Their new revisions had failed CPU-quota during deploy, and Cloud Run quietly kept routing traffic to the previous Ready revision (which is the right behaviour, but invisible from Firebase tooling). After ANY deploy with partial failures or transient errors, run: `gcloud run services describe <fnname-lowercase> --region=us-central1 --format="value(status.conditions[0].status, status.traffic[].revisionName)"` to verify `Ready=True` AND traffic is on the newest revision name. Lesson: "deploy complete" + green listing is not the same as "new code is serving traffic." Always cross-check the Cloud Run service state directly when stakes are high.
- [2026-05-19] When chasing the "outdated firebase-functions" deploy warning, the latest stable is v7.x — bumping just to v5.x clears the v4-deprecation warning but the CLI still flags v5 as outdated against v6/v7. Go to the latest major directly (admin v13, functions v7) unless there's a specific reason to stay on an LTS. `npm view firebase-functions version` shows latest.
